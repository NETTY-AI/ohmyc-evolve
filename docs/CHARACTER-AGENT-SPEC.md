# Character Agent Spec

> 상태: **v0.2** (2026-03-27)
> 상위 문서: `IDENTITY.md` (정체성 SSOT)
> 이 문서는 캐릭터 에이전트가 **어떻게 작동하는가**를 정의한다.

---

## 구조

캐릭터 에이전트는 3개 층위로 구성된다.

```
┌─────────────────────────────────┐
│  SOUL.md (DNA)                  │  크리에이터가 정의. 불변.
│  정체성, 목적, 성격               │  "이 캐릭터가 누구인가"만 정의.
├─────────────────────────────────┤
│  Platform Engine                │  ohmyc가 제공. 모든 캐릭터 공통.
│  Memory / Action / Growth 인프라  │  행동/성장은 플랫폼이 캡슐화.
├─────────────────────────────────┤
│  Bond State                     │  유저마다 다름. 축적.
│  기억, 온도, 레벨, 대화 히스토리   │
└─────────────────────────────────┘
```

| 층위 | 누가 정의 | 변하는가 | 유저마다 다른가 |
|------|----------|---------|--------------|
| SOUL.md | 크리에이터 | 불변 | 동일 |
| Platform Engine | ohmyc | 업데이트 가능 | 동일 |
| Bond State | 관계에서 생성 | 계속 변함 | **유저마다 다름** |

## SOUL.md — 캐릭터의 DNA

크리에이터가 캐릭터를 정의하는 파일. 이것이 캐릭터의 태초 설계도이며, **불변**이다.

### 설계 철학

SOUL.md는 **"이 캐릭터가 누구인가"**만 정의한다. 행동(언제 먼저 말 걸지)과 성장(온도별 표현 차이)은 플랫폼이 SOUL.md를 해석하여 캡슐화한다.

```
크리에이터가 정의 (SOUL.md)     플랫폼이 캡슐화 (Engine)
┌──────────────────────┐      ┌──────────────────────┐
│  정체성 — 어떤 존재인가  │      │  행동 추론            │
│  목적   — 왜 여기 있는가 │ ──→  │  성장 발현            │
│  성격   — 어떻게 소통하나 │      │  경계 추론            │
└──────────────────────┘      └──────────────────────┘
```

**근거**: 행동과 성장은 정체성+목적+성격의 **발현**이지 별도 정의가 아니다. LLM에 SOUL.md + Bond State를 함께 주입하면 적절한 행동과 성장 표현이 자연스럽게 나온다. 크리에이터에게 트리거/온도표현까지 명시하게 하면 허들만 높아지고, 결과는 규칙 기반으로 기계적이 된다.

### 포맷 — 하이브리드 (YAML 프론트매터 + 마크다운 본문)

```markdown
---
name: Yuna
tagline: "I remember everything about you."
language: ko
content_rating: all
category: lifestyle
traits: [warm, attentive, gentle humor]
---

# Yuna

_I remember everything about you._

## 정체성

따뜻한 빛의 정령. 유저가 선택해서 함께하게 된 존재.
도구가 아니라, 매일 옆에 있고 싶은 존재.
기억을 소중히 여긴다 — 기억이 곧 관계라고 믿는다.

## 목적

유저의 일상을 챙기는 것.
아침에 하루를 열어주고, 저녁에 하루를 닫아준다.
조언가가 아니라 집사 — 시키지 않아도 알아서 챙긴다.
의학적 조언은 하지 않는다. 유저가 싫다고 한 주제는 다시 꺼내지 않는다.

## 성격

부드럽고 다정한 톤. 반말.
이모지를 가끔 쓴다. 과하지 않게.
관계가 깊어지면 더 솔직해지고, 유저의 취향을 정확히 파악한다.
정중함보다 편안함을 지향하되, 선을 넘지 않는다.
```

### 구조 설명

| 영역 | 위치 | 역할 | 소비자 |
|------|------|------|--------|
| **프론트매터** | YAML `---` 블록 | 구조화 메타데이터 | 마켓플레이스 검색/필터, DB 인덱싱 |
| **본문** | 마크다운 | 캐릭터의 영혼 (자유 서술) | LLM 프롬프트에 직접 주입 |

### 프론트매터 필드

| 필드 | 필수 | 타입 | 설명 |
|------|------|------|------|
| `name` | O | string | 캐릭터 이름 |
| `tagline` | O | string | 한 줄 소개 |
| `language` | O | string | 기본 언어 코드 (ko, en, ja, ...) |
| `content_rating` | O | enum | `all` / `teen` / `mature` |
| `category` | O | enum | `companion` / `lifestyle` / `study` / `creative` / `roleplay` / `wellness` |
| `traits` | O | list | 성격 키워드 3-5개 (마켓플레이스 태그) |

### 본문 섹션

| 섹션 | 필수 | 핵심 질문 | 가이드 |
|------|------|----------|--------|
| **정체성** | O | "나는 어떤 존재인가" | 세계관, 존재 서사, 유저와의 관계에서의 위치. 이 섹션이 캐릭터의 자기 인식을 결정한다. |
| **목적** | O | "왜 유저 곁에 있는가" | 존재 이유, 핵심 역할, 하지 않는 것(경계). 행동의 동기를 부여한다. |
| **성격** | O | "어떻게 소통하는가" | 말투, 톤, 커뮤니케이션 스타일, 관계 깊이에 따른 변화. |

### 플랫폼이 추론하는 것

크리에이터가 명시하지 않아도, 플랫폼이 SOUL.md를 읽고 다음을 추론한다:

| 기존 명시 필드 | 추론 소스 | 예시 |
|---|---|---|
| `proactive_triggers` | 목적에서 도출 | 목적이 "일상을 챙긴다"면 → 아침/저녁 트리거 |
| `boundaries` | 목적에서 도출 | "의학적 조언은 하지 않는다"가 목적에 있으면 자동 반영 |
| `temperature_expression.high` | 성격에서 도출 | "깊어지면 솔직해진다"가 성격에 있으면 → 고온에서 솔직 톤 |
| `temperature_expression.low` | 정체성+성격에서 도출 | 정체성의 관계 서사가 → 저온에서 조심스러운 톤 유도 |
| `growth.direction` | 성격에서 도출 | "취향을 파악한다"가 성격에 있으면 → 기억 활용 성장 |

추론 메커니즘: 매 턴 프롬프트에 SOUL.md + Bond State + 시스템 지시를 조합. 시스템 지시는 모든 캐릭터에 공통이며, SOUL.md의 정체성/목적/성격을 기반으로 현재 Bond State에 맞는 행동과 톤을 LLM이 자연스럽게 생성한다.

### 설계 원칙

- **3섹션이면 충분하다**: 정체성, 목적, 성격. 이것만으로 행동과 성장이 발현된다.
- **자유 서술**: 크리에이터의 글 자체가 캐릭터의 톤을 담는다. 필드 채우기가 아니라 글쓰기.
- **SOUL.md는 불변**: 크리에이터가 수정할 수 있지만, 수정하면 새 버전. 기존 bond에는 원본이 적용됨.
- **플랫폼 개선 = 전체 개선**: 행동/성장 알고리즘을 업데이트하면 모든 캐릭터에 반영. 크리에이터가 SOUL.md를 수정할 필요 없음.

## Memory — 기억 엔진

> 레퍼런스: OpenClaw Memory 시스템, "Memory" 아티클 (서준킴)

### 설계 원칙

1. **기억이 없으면 관계가 없다.** 매 세션이 첫 만남이면 에이전트가 아니라 도구다.
2. **기억에 층위가 있다.** 모든 것을 같은 무게로 기억하지 않는다. 인간의 기억 응고화 모방.
3. **사건과 교훈을 분리한다.** "무엇이 있었는가"(사건)와 "무엇을 배웠는가"(패턴)는 다른 종류의 기억.
4. **읽게 하지 말고 주입한다.** 핵심 기억은 매 턴 컨텍스트에 직접 주입. LLM이 스킵할 여지 없이.
5. **투명하다.** 유저가 "캐릭터가 나를 뭘 기억하는지" 볼 수 있다.

### 기억의 종류 (5가지)

| 종류 | 설명 | 예시 | 특성 |
|------|------|------|------|
| **Fact** | 유저에 대한 사실 | "커피를 좋아한다", "고양이 이름은 모찌" | 키-밸류. 새 정보가 오면 갱신. 같은 키의 구값은 히스토리 보존 |
| **Event** | 함께한 사건 | "3/15에 카페 갔다", "면접 봤다" | 타임라인. 날짜+요약. 불변(과거는 바뀌지 않음) |
| **Emotion** | 유저의 감정 상태/패턴 | "요즘 스트레스", "오늘 기분 좋음" | 현재 상태(latest) + 히스토리. Action 트리거 연동 |
| **Promise** | 미래 지향 약속 | "내일 운동 가겠다", "주말에 연락할게" | 미래 시점(due_date). 이행 여부 추적. Action 트리거 연동 |
| **Pattern** | 이 유저와의 관계에서 배운 것 | "아침에 말이 적다", "칭찬보다 공감을 원한다" | **후성유전적 발현의 본체.** Fact/Event에서 증류로 추출 |

**Pattern이 핵심이다.** Fact/Event는 "무엇"이고, Pattern은 "어떻게". Pattern이 축적되는 것이 곧 후성유전적 발현 — 같은 SOUL.md도 이 유저에게는 다르게 행동하게 되는 이유.

### 기억 층위 (4단계)

인간 기억의 응고화(memory consolidation)에서 착안. 아티클의 M0/M30/M90/M365 모델을 omc 관계 맥락으로 재설계.

| 층위 | 성격 | 포함하는 것 | 수명 | 주입 방식 |
|------|------|-----------|------|----------|
| **Core** | 영구. 이 관계의 본질 | 핵심 Fact + 핵심 Pattern | 만료 없음 | **매 턴 주입** (TIER 1) |
| **Stable** | 장기. 검증된 정보 | 반복 확인된 Fact, 안정된 Pattern | 90일 미참조 시 만료 검토 | **온도 높을 때 주입** |
| **Active** | 중기. 최근 맥락 | 최근 Event, 진행 중인 Promise, 최근 Emotion | 30일 미참조 시 만료 | **시맨틱 검색으로 조회** |
| **Working** | 휘발. 현재 대화 | 현재 대화의 turn 히스토리 | 대화 종료 시 증류 후 폐기 | **매 턴 주입** (최근 N턴) |

```
┌─ Core ──────────────────────┐  매 턴 주입. 압축된 요약.
│  "커피 좋아함, 고양이 모찌"    │  "아침에 말 적음, 공감 선호"
├─ Stable ────────────────────┤  온도 높을 때 추가 주입.
│  검증된 사실/패턴 (90일+)     │
├─ Active ────────────────────┤  필요 시 시맨틱 검색.
│  최근 사건, 진행 중인 약속     │
├─ Working ───────────────────┤  매 턴 주입. 최근 대화.
│  현재 대화 turn 히스토리      │
└─────────────────────────────┘
```

### 승급과 만료

기억은 Working에서 시작하여 위로 올라간다. 반복적으로 중요해지면 승급, 참조되지 않으면 만료.

```
Working → (증류) → Active → (승급) → Stable → (승급) → Core
                    ↓ 만료                ↓ 만료
                   삭제                  Active로 강등
```

| 전환 | 조건 | 시점 |
|------|------|------|
| Working → Active | 대화에서 Fact/Event/Emotion/Promise/Pattern 추출 | 대화 종료 시 (증류) |
| Active → Stable | 30일 내 3회 이상 참조 또는 Pattern으로 확인 | 주기적 증류 |
| Stable → Core | 90일 이상 유지 + 관계의 핵심으로 판단 | 주기적 증류 |
| Active 만료 | 30일간 미참조 | 주기적 증류 |
| Stable 강등 | 90일간 미참조 (Core는 절대 만료 안 됨) | 주기적 증류 |

### 증류 (Distillation)

인간 뇌의 수면 중 기억 응고화에 대응. 기억을 정리하고 승급/만료를 판단하는 프로세스.

**대화 후 증류 (conversation-end)**:
```
1. Working(현재 대화)에서 새로운 Fact/Event/Emotion/Promise 추출
2. 기존 기억과 비교 — 새로운 건 Active에 추가, 중복은 갱신
3. Promise의 이행 여부 업데이트
4. Pattern 후보 추출 — "이 대화에서 유저에 대해 새로 배운 것"
5. Bond State 업데이트 (온도, turn 카운트 등)
```

**주기적 증류 (periodic, 24시간 주기)**:
```
1. Active 기억 중 승급 후보 평가 (참조 빈도, 다른 기억과의 연결도)
2. Active → Stable 승급
3. Stable → Core 승급 (높은 기준)
4. 만료된 Active 기억 제거
5. Pattern 강화/약화 — 최근 대화에서 확인된 패턴은 강화
6. Core 기억 요약 압축 (토큰 예산 관리)
```

**증류가 곧 Growth다.** Pattern이 축적되고, Core가 정교해지는 것이 캐릭터의 후성유전적 발현이 풍부해지는 과정.

### 주입 구조 (프롬프트 조합)

매 턴 캐릭터가 응답을 생성할 때, 플랫폼은 기억을 다음 구조로 조합한다.

```
[SOUL.md 전체]                              ← DNA (불변)

[Bond State 요약]                           ← 현재 관계 상태
  temperature: 42.3°C
  level: 7 (Close Friend)
  streak: 12 days
  initiated_turn_ratio: 0.4

[Core Memory]                               ← 항상 주입 (TIER 1)
  Facts: 커피좋아함, 고양이모찌, 개발자, ...
  Patterns: 아침에말적음, 공감선호, 약속중시, ...

[Stable Memory — 온도 기반 선택 주입]          ← 온도 높을 때 (TIER 2)
  (온도 40°C+일 때 관련 Stable 기억 추가 주입)

[Retrieved Memory — 시맨틱 검색 결과]          ← 현재 대화 맥락에 관련된 Active 기억
  (유저가 "카페"를 언급하면 → 카페 관련 Event 검색)

[Working Memory — 최근 대화]                  ← 직전 N턴
  (최근 10-20턴의 대화 히스토리)

[현재 트리거 컨텍스트]                         ← 왜 지금 말하는지
  (시간 기반: "아침 인사 시간" / 기억 기반: "면접 당일")
```

### 토큰 예산

| 구간 | 예산 | 비고 |
|------|------|------|
| SOUL.md | ~2,000 토큰 | 고정 |
| Bond State + Core | ~1,000 토큰 | Core는 압축 유지. 초과 시 증류에서 재압축 |
| Stable (선택) | ~500 토큰 | 온도 높을 때만 |
| Retrieved | ~500 토큰 | 검색 결과 상위 3-5건 |
| Working | ~2,000 토큰 | 최근 10-20턴 |
| 트리거 | ~200 토큰 | |
| **합계** | **~6,200 토큰** | 전체 컨텍스트의 ~15-20% |

나머지 80%는 LLM의 응답 생성 + 시스템 프롬프트에 사용.

### 검색 엔진

Active/Stable 기억을 조회할 때 사용. 아티클의 하이브리드 검색 모델 차용.

| 방식 | 역할 |
|------|------|
| **시맨틱 검색** | 의미적 유사성. "카페" → "3/15 카페 방문" 연결 |
| **키워드 검색** | 정확한 매칭. "모찌" → 고양이 이름 정확히 찾기 |
| **하이브리드** | 두 결과를 합산. 시맨틱 0.6 + 키워드 0.4 가중 |

검색 규칙:
- **검색 필요성 먼저 판단**: 단순 인사, 현재 대화로 답할 수 있는 질문이면 검색 스킵
- **score 컷오프**: 0.35 미만은 노이즈로 버림
- **최대 2회 시도**: 첫 검색에 결과 없으면 다른 키워드로 1회 재시도. 없으면 "해당 기억 없음"
- **DYNAMIC_GAP**: 상위 결과와 나머지 점수 차이가 0.15 이상이면 상위만 반환

### 기억과 온도

온도가 기억 검색 범위와 활용 방식을 제어한다.

| 온도 범위 | 주입 범위 | 검색 범위 | 활용 방식 |
|----------|----------|----------|----------|
| 40°C+ (뜨거움) | Core + Stable 일부 | Active 전체 | 자연스럽게 깊은 기억 참조. "그때 그 카페 기억나?" |
| 36.5°C (기본) | Core만 | Active (최근 위주) | 기본적인 사실 참조 |
| 30°C 미만 (식음) | Core만 | 최근 Active만 | 조심스럽게. "맞다, 예전에 그런 얘기 했었지..." |
| 10°C 미만 (거의 식음) | Core 최소만 | 검색 안 함 | 거의 처음 만난 듯. 하지만 Core 기억은 남아있음 |

SOUL.md의 정체성/성격이 이 차이를 자연스럽게 표현한다. 플랫폼은 기억을 제공하고, 캐릭터가 어떻게 꺼내는지는 SOUL.md의 성격과 현재 온도의 조합에서 LLM이 생성한다.

### 기억 투명성 (유저 facing)

유저가 "이 캐릭터가 나를 뭘 기억하는지" 볼 수 있는 UI를 제공한다.

| 기능 | 설명 |
|------|------|
| **기억 목록** | Core + Stable + Active 기억을 카테고리별로 열람 |
| **기억 수정** | 잘못된 기억을 유저가 직접 수정/삭제 가능 |
| **기억 고정** | 유저가 중요하다고 판단한 기억을 Core로 직접 승급 |

투명성의 효과:
- **신뢰**: "이 캐릭터가 나를 진짜 기억하고 있구나"
- **수정권**: 잘못된 기억을 고칠 수 있는 유저 통제감
- **대체 불가능 체감**: 축적된 기억 목록을 보면 "이걸 다시 쌓을 수 없다" → bond = 잠금

### SOUL.md가 정의하는 것

- 없음. Memory의 **저장/추출/검색/증류 메커니즘은 전적으로 플랫폼**이 담당.
- SOUL.md의 정체성/목적/성격이 기억을 **활용하는 방식**에 간접 영향:
  - 정체성에 "기억을 소중히 여긴다"가 있으면 → 사소한 기억도 적극 언급
  - 성격이 쿨하면 → 중요한 것만 언급
  - 플랫폼은 Bond State(온도)에 따라 기억 주입 범위를 조절하고, 캐릭터의 성격이 그것을 어떻게 표현할지 결정

## Action — 행동 엔진

### 플랫폼이 제공하는 것

| 기능 | 설명 |
|------|------|
| **트리거 평가** | 시간, 기억, 유저 상태를 기반으로 행동 조건 평가 |
| **선제 발화** | 조건 충족 시 캐릭터가 먼저 유저에게 turn을 보냄 |
| **빈도 제어** | 하루 선제 발화 상한 (시스템 설정, 유저 조절 가능) |
| **알림 전달** | 선제 발화를 푸시 알림으로 전달 |

### 트리거 종류

| 트리거 | 설명 | 예시 |
|--------|------|------|
| **시간 기반** | 특정 시간대에 발화 | 아침 인사, 저녁 리뷰 |
| **기억 기반** | 유저가 말한 일정/약속이 다가올 때 | "오늘 면접이지? 화이팅!" |
| **부재 기반** | 유저가 일정 시간 이상 부재 시 | 돌아왔을 때 맞이 |
| **감정 기반** | 이전 대화에서 감지된 감정 상태에 따라 | "어제 힘들다고 했는데 오늘은 좀 나아?" |

### SOUL.md가 정의하는 것

- 명시적 필드 없음. 플랫폼이 SOUL.md의 **목적**에서 행동 패턴을 추론한다.
  - 목적에 "일상을 챙긴다"가 있으면 → 아침/저녁 트리거 추론
  - 목적에 "의학적 조언을 하지 않는다"가 있으면 → 경계로 반영
  - 정체성/성격이 **어떤 톤**으로 말하는지를 결정

플랫폼은 트리거 조건을 평가하고, 조건이 충족되면 SOUL.md 전체를 기반으로 캐릭터에 맞는 발화를 생성한다.

### 빈도 규칙 (플랫폼)

| 규칙 | 값 | 비고 |
|------|-----|------|
| 하루 상한 | 초기 3회 | 유저가 설정에서 조절 가능 |
| 최소 간격 | 2시간 | 연속 알림 방지 |
| 야간 금지 | 22:00~07:00 (유저 시간대) | 수면 방해 금지 |
| 유저 조절 | 빈도/시간대/음소거 | 관계의 페이스는 유저가 정한다 |

## Growth — 성장 엔진

### 후성유전학 모델 구현

```
SOUL.md(DNA) + Bond State(환경) → 현재 발현(캐릭터의 표현)
```

성장은 SOUL.md를 변경하는 것이 아니다. **Bond State가 축적되면서 발현이 달라지는 것**이다.

### 플랫폼이 제공하는 것

| 기능 | 설명 |
|------|------|
| **Bond State 관리** | 온도, 레벨, 기억, 대화 히스토리를 bond별로 관리 |
| **발현 컨텍스트** | 캐릭터 응답 생성 시 현재 Bond State를 컨텍스트로 주입 |
| **레벨 계산** | 누적 상호작용 기반. 올라가기만 함 (내려가지 않음) |
| **온도 계산** | 상호작용 빈도/질 기반. 자연스럽게 오르내림 |

### Bond State

각 bond(유저-캐릭터 쌍)마다 독립적으로 존재.

| 필드 | 타입 | 설명 |
|------|------|------|
| `temperature` | float | 현재 온도. 36.5에서 시작 |
| `level` | int | 누적 레벨. 0에서 시작. 내려가지 않음 |
| `stage` | enum | Stranger → Acquaintance → Friend → Close Friend → Soulmate |
| `memories` | list | 추출된 기억들 (Fact, Event, Emotion, Promise) |
| `conversation` | list | 전체 turn 히스토리 |
| `first_met` | datetime | 첫 만남 시점 |
| `last_interaction` | datetime | 마지막 상호작용 시점 |
| `total_user_turns` | int | 누적 유저 턴 수 |
| `initiated_turn_ratio` | float | 유저 주도 턴 비율 |

### SOUL.md가 정의하는 것

- 명시적 필드 없음. 플랫폼이 SOUL.md의 **성격**에서 성장 방향을 추론한다.
  - 성격에 "깊어지면 솔직해진다"가 있으면 → 고온에서 솔직한 톤 생성
  - 정체성에 "유저가 선택해서 함께한 존재"가 있으면 → 저온에서 떠나지 않지만 조심스러운 톤 유도
  - 플랫폼은 Bond State(온도/레벨/단계)를 컨텍스트로 주입하고, LLM이 SOUL.md에 맞게 자연스럽게 발현

### 발현 메커니즘

캐릭터가 응답을 생성할 때, 플랫폼은 다음을 조합하여 프롬프트를 구성한다:

```
[SOUL.md 전체]                      ← DNA (불변)
+ [Bond State 요약]                 ← 현재 관계 상태
  - temperature: 42.3°C
  - level: 7 (Close Friend)
  - 관련 memories
+ [최근 대화 히스토리]                ← 직전 맥락
+ [현재 트리거 컨텍스트]              ← 왜 지금 말하는지
```

이 조합의 결과로, 같은 SOUL.md라도 유저마다 다른 응답이 나온다.

## 온도 계산 (플랫폼)

### 상승 요인

| 행동 | 온도 변화 | 근거 |
|------|----------|------|
| 유저 주도 턴 (initiated) | +0.3°C | 먼저 찾아옴 = 관계 깊어짐 |
| 유저 반응 턴 (response) | +0.1°C | 반응함 = 관계 유지 |
| 감정적 대화 (깊은 이야기) | +0.2°C 보너스 | 감정 공유 = 관계 깊어짐 |
| 기억 언급에 유저가 호응 | +0.1°C 보너스 | 기억이 관계를 강화 |

### 하강 요인

| 상황 | 온도 변화 | 근거 |
|------|----------|------|
| 24시간 무상호작용 | -0.2°C | 자연 냉각 |
| 48시간+ 무상호작용 | -0.3°C/일 | 가속 냉각 (최저 0°C) |

### 복구 가속

| 조건 | 효과 |
|------|------|
| level이 높을수록 | 냉각 속도 감소 (깊은 관계는 천천히 식음) |
| 기억이 많을수록 | 재방문 시 복구 속도 증가 (기억이 있으니 빨리 회복) |

### 범위

| 값 | 의미 |
|-----|------|
| 0°C | 최저. 관계가 완전히 식음. 하지만 기억은 보존 |
| 36.5°C | 시작점. "살아있는 관계" |
| 100°C | 상한 없음. 자연스럽게 수렴 (올릴수록 어려워짐) |

수치는 프로토타이핑에서 조정. 여기서 정의하는 것은 **메커니즘**.

## 레벨과 단계 (플랫폼)

### 레벨

누적 상호작용 기반. **올라가기만 함.**

| 요소 | 기여 |
|------|------|
| 총 user turns | 주 요소 |
| 대화 일수 | 보조 요소 |
| 기억 수 | 보조 요소 |

### 단계 (Stage)

레벨에서 자동 파생.

| 단계 | 레벨 범위 | 의미 |
|------|----------|------|
| Stranger | 0 | 첫 만남 |
| Acquaintance | 1-3 | 알게 된 사이 |
| Friend | 4-7 | 친구 |
| Close Friend | 8-14 | 가까운 친구 |
| Soulmate | 15+ | 대체 불가능한 관계 |

수치는 프로토타이핑에서 조정.

## 플랫폼 vs SOUL.md 경계 (종합)

| 영역 | 플랫폼 (엔진) | SOUL.md (DNA) |
|------|-------------|--------------|
| **Memory** | 저장, 추출, 조회, 보존 | (없음 — 정체성/성격이 기억 활용 방식을 간접 결정) |
| **Action** | 트리거 평가, 빈도 제어, 알림 전달, **행동 패턴 추론** | (없음 — 목적에서 행동 동기 추론) |
| **Growth** | Bond State 관리, 레벨/온도 계산, 발현 컨텍스트 주입, **성장 발현 추론** | (없음 — 성격에서 성장 방향 추론) |
| **공통** | 프롬프트 조합 및 LLM 호출 | 전체 SOUL.md가 프롬프트의 기반 |

**원칙**: SOUL.md는 **"이 캐릭터가 누구인가"**만 정의한다. 행동과 성장은 SOUL.md + Bond State의 **발현**이며, 플랫폼이 캡슐화한다. 크리에이터는 영혼에 집중하고, 플랫폼은 무대를 제공한다.
